# Game Scope - Iteration & MVP

O escopo "final" de um jogo raramente é o que foi escrito no primeiro dia. Ele deve ser flexível para sobreviver ao contato com os jogadores.

## 1. O Escopo do Soft Launch / Demo
Você consegue fechar um pacote funcional que represente o jogo sem ter 100% pronto?
- **Exemplo**: Lançar apenas o "Ato 1" ou "Mundo de Tutorial".
- Isso permite testar a **Retenção** (o jogador volta no dia seguinte?).
- Se a retenção for baixa, o escopo dos mundos futuros deve ser repensado imediatamente.

## 2. Reserva de Feedback
Não gaste 100% dos seus recursos no planejamento inicial. Reserve 20% do escopo para:
- Corrigir o que os jogadores acharam chato.
- Adicionar pequenas features que os jogadores pediram.
- Ajustar a dificuldade baseada em dados reais.

## 3. O MVP como Fundamento
Lembre-se: O propósito do escopo inicial é chegar ao **MVP** o mais rápido possível.
- O escopo além do MVP deve ser visto como "Carga Extra" que pode ser descartada se o tempo apertar.

## Conclusão
Um desenvolvedor inteligente sabe quando o escopo deve ser "congelado" para o lançamento e quando ele deve ser "derretido" para ajustes baseados em feedback. Escopo é um organismo vivo.
